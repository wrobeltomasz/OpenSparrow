OpenSparrow — Admin Panel Overview
==================================
The Admin panel (/admin/) is where the entire OpenSparrow application is configured:
tables, forms, dashboards, calendars, workflows, automations, users, and system maintenance.
Available to the admin role (and partially to editor). All configuration is stored as JSON
files in config/ — no code changes required.

Current version: 3.0.
Stack: PHP 8.4+, PostgreSQL 14+, vanilla JavaScript (ES6+). No npm, no composer required.
License: LGPL v3. Self-hosted, works fully offline.


SETUP & INSTALLATION
--------------------
1. Open the app — the first-run setup wizard (setup.php) configures the database connection.
   (Alternatively copy config/database.json.example to config/database.json manually.)
2. Open /admin — click System → Database → Initialize System Tables.
3. Log in as admin with the randomly generated password shown once in the setup wizard —
   change it after first login.
Docker: docker compose up -d --build → http://localhost:8080
System tables live in the PostgreSQL schema from database.json (default "app"),
all prefixed spw_.


Navigation sections: Overview, Data Management, Workflows, Knowledge Base, System,
Configuration. A Docs button in the header opens built-in searchable documentation
(Getting Started / Schema / Grid Features / Admin Panel / Troubleshooting).


OVERVIEW (landing tab)
----------------------
Dashboard shown on entering the admin panel: application version, database size,
recent record activity, and recent cron runs.


DATA MANAGEMENT
---------------
Schema:
  - Manage tables and columns ("schema" config key)
  - Global settings: default grid page size, visibility options
  - Per table: display name, icon, hidden flag (excludes from menu), default sort,
    initial load limit
  - Per column: name, type (string / int / enum / FK / M2M / file / virtual), display
    name, description (grid header tooltip), nullable, unique, readonly, default value,
    regex validation with custom error message, enum options with colors, FK config
    (referenced table + column + display column)
  - Drag-and-drop column reordering
  - Subtable configuration (FK back-references shown inside edit forms)
  - M2M (many-to-many) configuration

Add Table:
  - Wizard for creating a new PostgreSQL table and registering it in the schema

Dashboard:
  - Add widgets: stat card (COUNT/SUM/AVG aggregate), bar chart, pie chart, table (recent records list)
  - Per widget: title, source table, aggregate (COUNT/SUM/AVG/MIN/MAX), axis columns,
    group-by, filter conditions, width/height, color palette
  - Drag widgets to reorder in a 3-column grid; live preview while editing

Calendar:
  - Configure event sources: table + date/timestamp column
  - Event title template (which column becomes the title)
  - Color rules (per enum value or fixed hex), optional icon
  - Advance notification days (notify_before_days) per source

CSV Import:
  - Upload CSV, map columns to table fields, preview, import in batches
  - Import history tracked in spw_csv_import_log

Files:
  - File management settings: allowed types, size limits ("files" config key)

Menu Preview:
  - Sidebar navigation structure ("menu" config key): visibility, order, grouping

User Records:
  - Configures the "My records" panel every user opens from the avatar menu
    ("user_records" config key)
  - Column Mapping tab: per table, which columns are concatenated into each
    assigned record's label
  - Global Settings tab: how many recently-assigned records are shown per table

Schema Map (ERD):
  - Visual entity-relationship diagram: all tables with FK relations drawn as arrows
  - Click a table to jump to its schema editor; zoom and pan controls

Views:
  - Read-only PostgreSQL views exposed as grid pages ("views" config key)
  - Create/edit/delete view definitions: name, description, filters, column order
  - Drill-down by group_by levels (e.g. Year → Month → Invoice)
  - Sync discovers views (and column metadata) in the searched schema(s); a Schemas
    tab selects which PostgreSQL schemas beyond the default app schema are searched

Board (Kanban):
  - Configure a source table, a status column (its enum options/colors become the
    board's lanes), card title column, and detail fields shown on each card ("board"
    config key)
  - Board appears in the sidebar menu only once a table and status column are set

Print Templates:
  - Printable report templates bound to a PostgreSQL view ("print" config key)
  - Block editor: heading, text, and table blocks with {column} variables pulled
    from the bound view; optional report parameters (dropdown filters, e.g. by
    employee) resolved either from the column's distinct values or a lookup view
  - Rendered at print.php with automatic A4 pagination and a dedicated print
    stylesheet; saved/validated via api/print.php


WORKFLOWS
---------
Workflow Manager:
  - Guided multi-step entry flows shown to users (e.g. Company → Contact → Deal)
  - Per workflow: title, description, icon
  - Step builder: add / remove / drag-reorder steps
  - Per step: target table, form fields, multi-record option (e.g. several contacts)

Automations:
  - Event-driven rules executed by the backend on record events
  - Rule list shows name, table, event, status; Edit / History / Delete per rule
  - Triggers: After create (INSERT), After update (PATCH)
  - Conditions: AND/OR groups, nestable; operators equals, not equals, contains,
    not contains, is empty, is not empty
  - Actions:
      1. Update fields on this record
      2. Send notification — recipient user ID, title, optional link; one notification
         per (rule, record, user, day), duplicates ignored
      3. Create record in another table — target table + field mapping
  - Values support variables {{ record.field_name }} and {{ current_user.id }}
  - Run History: last 100 executions per rule — time, table, record ID, event,
    status (ok / error / skipped), error message; stored in spw_automation_runs


KNOWLEDGE BASE
--------------
RAG Documents:
  - Upload .txt documents (drag & drop) with comma-separated tags; convert PDF/Word
    to plain text before uploading
  - File list: name, size, tags, date; delete, retag, search by name or tag
  - Statistics: total files and tokens, per-file token counts, query history
  - Configure Ollama (URL, model) and test retrieval-augmented queries directly
  - Retrieval uses PostgreSQL full-text search; tag filtering narrows the search


SYSTEM
------
Audit & Snapshots:
  - Log of all record mutations (INSERT / UPDATE / DELETE): timestamp, user, action,
    table, record ID
  - Filter by table, user, action type; view before/after record snapshots

Backup Tables:
  - Manual backup button (SQL dump), list of backups with timestamps
  - Download backup (.sql file); restore from backup (confirmation required)

Cron Notifications:
  - Manual run (trigger the notification cron immediately)
  - Run history (status, output, duration), statistics (avg duration, success rate)
  - Setup guide for the server system cron (cron/notifications.php, e.g. every 15 min)
  - Log cleanup (delete old cron log entries)

Database:
  - Initialize System Tables (creates/updates spw_* tables — required on first install)
  - PostgreSQL schema browser, table creation wizard, column type documentation

Demo Systems:
  - One-click install of a pre-built demo with its own PostgreSQL schema, sample data,
    dashboard widgets, calendar sources, workflows, and views
  - Demo: CRM (companies, contacts, deals, activities, leads, products,
    quotes, invoices, assets) — installs into its own spw_crm schema
  - Reset button removes the demo data and its configuration entries

Health Check:
  - Database connection status, table bloat analysis, index advisor,
    slow query log, backup status

M2M Builder:
  - Visual builder for many-to-many relations: left/right FK tables, junction table
    name, display columns

ETL:
  - Scheduled or on-demand batch import from one or more external source databases
    (MySQL, MariaDB, PostgreSQL, SQLite) or a CSV file fetched from an FTP/FTPS
    server, into local PostgreSQL target tables (driver-based, extensible to
    Oracle/DB2/SQL Server); independent of the removed MySQL Gateway feature
  - Sources tab: define 2+ named source connections (name, driver, host, port,
    database, user, password), each with its own Test connection. A "CSV file
    (FTP/FTPS)" source instead configures protocol (FTP or FTPS), remote directory,
    CSV file name, column delimiter, whether the first row is a header, and passive
    mode — no true SSH/SFTP support (requires ext-ssh2/phpseclib, not bundled)
  - Jobs: each job picks one of the configured sources. Database-source jobs supply
    a read-only source SELECT/WITH query; CSV/FTP-source jobs import the whole file
    on every run (no query). The target is a schema picked from a dropdown (any
    non-system PostgreSQL schema) followed by a table dropdown populated
    dynamically from that schema's tables (spw_* system tables excluded). All jobs
    share an optional column map (source column → target column) and a load
    mode — full refresh (truncate+insert), append, or upsert (ON CONFLICT)
  - Incremental loading via a {{watermark}} placeholder substituted with the last
    seen value of an incremental column (database sources only)
  - Preview shows sample rows before saving; Test connection verifies reachability
    (for CSV/FTP sources, also confirms the configured file exists)
  - Schedule: manual / daily / weekly / monthly, run by cron/cron_etl.php; run
    history (rows read/written, duration, errors) in spw_etl_log
  - Flows tab: chains existing ETL jobs into an ordered sequence via a tile-based
    builder (start, job, job, ..., end tiles; steps reference job ids from the Jobs
    tab and can be reordered/removed). Runs strictly in order and stops immediately
    at the first failing step. Same manual/daily/weekly/monthly schedule model,
    guarded per flow, run by cron/cron_etl_flow.php; run history (overall status,
    failed step) in spw_etl_flow_run_log plus per-step detail in
    spw_etl_flow_step_log

Anonymization (GDPR):
  - Rules per column: table, a date/timestamp column, an age threshold (anonymize
    records older than N days), the PII column, and a replacement value
  - Preview (dry run) counts affected rows without modifying data
  - Schedule: manual / daily / weekly / monthly via cron/cron_anonymization.php
  - Scan Schema suggests PII columns by matching column names against a keyword
    dictionary; run history in spw_anonymization_log

Migrations:
  - Database migrations tab: list of system migrations (pending / applied),
    "Apply pending migrations" button
  - Release migrations tab: config cleanup driven by config/migrations.json —
    tracks files and config keys removed between versions, applied manually on upgrade

Performance:
  - 6 diagnostic sections, each with a Scan button: index advisor (missing indexes with
    recommended SQL), unused indexes, slow query log, table bloat, DB health (connection
    pool, cache hit rate), schema warnings (e.g. table without primary key)
  - Copy SQL buttons for recommended fix queries

Settings:
  - Application settings: system name, theme, menu configuration, notification settings
  - Custom Logo: replace the default frontend header logo (PNG/JPEG/WEBP, up to 2 MB);
    independent show/hide toggle
  - Security configuration stored in config/security.json: CORS, HSTS, CSP modes,
    optional IP whitelist, session timeout, password policy

Users:
  - User list: ID, name, status (active/inactive), role
  - Role dropdown per user: admin / editor / viewer
  - Activate / deactivate accounts; per-user password change dialog with strength meter
  - Add new user form: name + password + role


CONFIGURATION
-------------
Run Notifications Cron — trigger the notification worker manually.


CONFIGURATION STORE (spw_config table)
--------------------------------------
Application configuration lives in the spw_config table (one JSONB row per key,
optimistic locking via a version column, full change history in spw_config_log).
Read and written through includes/config_store.php — never as files.

  schema        — table/column definitions
  menu          — sidebar navigation structure
  settings      — app name, locale, feature toggles
  dashboard     — widget layout
  calendar      — calendar event sources
  board         — Kanban board (table, status column, card columns)
  workflows     — workflow step definitions
  automations   — automation rules (triggers, conditions, actions)
  views         — view definitions
  files         — file upload settings (allowed types, size limits)
  print         — printout templates
  anonymization — GDPR anonymization rules and dictionary
  user_records  — per-table owner column mapping and record limits
  rag           — Ollama URL, model, max context files, file size limit, timeout
  etl           — external source connection and import job definitions
  etl_flows     — ordered chains ("flows") of existing ETL job ids, run
                  sequentially, stopping on the first step failure

CONFIGURATION FILES (config/*.json)
------------------------------------
Only bootstrap and distribution files remain on disk:

  database.json       — DB connection (host, port, dbname, user, password, schema);
                        read before any database connection exists, so it cannot live in the DB
  security.json       — CORS, HSTS, CSP, IP whitelist, session and password policy
  mysql_connection.json — MySQL source connection credentials (ETL module)
  migrations.json     — release manifest (files/config keys removed per version)
  demo_meta.json      — installed demo metadata
